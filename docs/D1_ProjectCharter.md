โครงการระบบลงทะเบียนอบรม / Workshop ออนไลน์ของมหาวิทยาลัย
Project Charter
1. ที่มาและความสำคัญของโครงการ
ในปัจจุบัน มหาวิทยาลัยมีการจัดกิจกรรมอบรมและ Workshop จำนวนมาก ทั้งในด้านวิชาการ ทักษะวิชาชีพ และกิจกรรมเสริมหลักสูตร อย่างไรก็ตาม กระบวนการลงทะเบียนเข้าร่วมกิจกรรมส่วนใหญ่มักดำเนินการผ่านช่องทางที่ไม่เป็นระบบ เช่น การลงทะเบียนผ่านแบบฟอร์มออนไลน์ทั่วไปหรือการลงทะเบียนหน้างาน ส่งผลให้เกิดปัญหาด้านการจัดการข้อมูลผู้เข้าร่วม ความซ้ำซ้อนของข้อมูล และความยากลำบากในการสรุปรายงานผล
โครงการนี้จึงมีเป้าหมายเพื่อพัฒนาระบบลงทะเบียนอบรม / Workshop ออนไลน์ที่ช่วยให้การจัดการข้อมูลเป็นระบบ ลดข้อผิดพลาด และเพิ่มความสะดวกทั้งต่อผู้เข้าร่วมกิจกรรมและผู้จัดอบรม

2. วัตถุประสงค์ของโครงการ
-เพื่อออกแบบและพัฒนาระบบลงทะเบียนอบรม / Workshop ออนไลน์ที่ใช้งานง่ายและเป็นระบบ
-เพื่ออำนวยความสะดวกให้ผู้เข้าร่วมสามารถสมัคร ตรวจสอบสถานะ และยกเลิกการสมัครได้ด้วยตนเอง
-เพื่อช่วยให้ผู้จัดอบรมสามารถจัดการข้อมูล Workshop และรายชื่อผู้เข้าร่วมได้อย่างมีประสิทธิภาพ
-เพื่อสนับสนุนการจัดทำรายงานและสถิติที่เกี่ยวข้องกับการจัดอบรมของมหาวิทยาลัย

3. ขอบเขตของโครงการ (Project Scope)
3.1 ขอบเขตที่อยู่ในโครงการ (In Scope)
-ระบบแสดงรายการอบรม / Workshop
-ระบบสมัครและยกเลิกการสมัครอบรม
-การจำกัดจำนวนผู้เข้าร่วม Workshop
-ระบบจัดการ Workshop สำหรับผู้จัดอบรม
-ระบบอนุมัติ Workshop โดยผู้ดูแลระบบ
-ระบบแสดงและส่งออกรายชื่อผู้เข้าร่วม
-ระบบรายงานสรุปข้อมูลการอบรม
3.2 ขอบเขตที่อยู่นอกโครงการ (Out of Scope)
-ระบบชำระเงินออนไลน์
-ระบบออกใบประกาศนียบัตรอัตโนมัติ
-ระบบถ่ายทอดสดหรือเรียนออนไลน์
-ระบบเชื่อมต่อกับฐานข้อมูลภายนอกเชิงลึก

4. ผู้มีส่วนเกี่ยวข้องและบทบาท (Stakeholders & Roles)
Product Owner (PO): กำหนดความต้องการ ขอบเขต และลำดับความสำคัญของระบบ
Scrum Master (SM): ดูแลกระบวนการทำงานตาม Scrum
Developer (DEV): ออกแบบและพัฒนาระบบ
QA / DevOps: ทดสอบระบบและควบคุมคุณภาพ
ผู้ใช้ระบบ: นักศึกษา อาจารย์ และผู้จัดอบรม

5. ความเสี่ยงของโครงการและแนวทางจัดการ
ลำดับ     ความเสี่ยง                            แนวทางจัดการ
1.       ขอบเขตงานขยายเกินแผน                กำหนด MVP และควบคุม Scope
2.       เวลาพัฒนาไม่เพียงพอ                   วางแผน Sprint ชัดเจน
3.       ความเข้าใจ Requirement ไม่ตรงกัน       Review User Story ทุก Sprint
4.       ระบบซับซ้อนเกินความจำเป็น              ลดฟีเจอร์ที่ไม่จำเป็น
5.       การสื่อสารในทีมไม่ทั่วถึง                 จัดประชุม Scrum อย่างสม่ำเสมอ

6. แผนดำเนินงานโดยสังเขป (High-level Timeline)
Sprint 1: วิเคราะห์ความต้องการและออกแบบระบบ
Sprint 2: พัฒนาฟังก์ชันหลักของระบบ
Sprint 3: ทดสอบ ปรับปรุง และสรุปผลโครงการ
